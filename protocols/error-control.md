# 差错控制在网络模型中的层级分布

## 概述

差错控制（Error Control）贯穿整个协议栈，但**每一层的职责边界与策略截然不同**。核心原则：

> **L1 纠错（FEC）→ L2 检错丢包（CRC）→ L3 只检头部（Header Checksum）→ L4 端到端可靠（TCP）/ 可选校验（UDP）→ L7 应用级校验与重试**

```
┌─────────────────────────────────────────────────────────────┐
│ 应用层 (L7)   HTTP Retry · DNS Retransmit · MD5/CRC32 · ETag│  应用级校验/重试
├─────────────────────────────────────────────────────────────┤
│ 传输层 (L4)   TCP: CRC+SEQ+ACK+SACK+RTO+FastRetransmit     │  端到端可靠
│               UDP: 可选校验和 (不重传)                       │
├─────────────────────────────────────────────────────────────┤
│ 网络层 (L3)   IPv4 头部校验和 · IPv6 移除（依赖上下层）     │  只检头部
│               ICMP 错误报告                                 │
├─────────────────────────────────────────────────────────────┤
│ 链路层 (L2)   Ethernet FCS (CRC-32) · HDLC ARQ · Wi-Fi ACK │  单跳检错/重传
├─────────────────────────────────────────────────────────────┤
│ 物理层 (L1)   FEC (Reed-Solomon / KP4) · 交织 · 均衡       │  比特级纠错
└─────────────────────────────────────────────────────────────┘
```

## 三大基本机制

### 1. 检错（Error Detection）
| 机制 | 原理 | 典型应用 |
|------|------|----------|
| **奇偶校验** | 1 bit 使 1 的个数为偶/奇 | 老式串口、内存 ECC 辅助 |
| **校验和 (Checksum)** | 字节累加取反码 | IP/TCP/UDP |
| **CRC (循环冗余校验)** | 多项式除法余数 | 以太网 FCS (CRC-32)、Wi-Fi、HDLC |
| **密码学哈希 (HMAC)** | 密钥 + 哈希 | IPsec ESP 认证、TLS |
| **AEAD (GCM/CCM)** | 加密 + 认证一体 | TLS 1.3、MACsec、QUIC |

### 2. 纠错（Error Correction）
| 机制 | 原理 | 典型应用 |
|------|------|----------|
| **汉明码 (Hamming)** | 插入校验位定位错误 | ECC 内存 |
| **Reed-Solomon (RS)** | 多项式码，可纠 t 个符号 | 100G/400G 以太网 RS-FEC |
| **卷积码 + Viterbi** | 状态机编码 | Wi-Fi、卫星、5G |
| **LDPC** | 稀疏校验矩阵 | Wi-Fi 6 (802.11ax)、5G |
| **Turbo 码** | 并行卷积码 | 4G LTE |
| **KP4 FEC** | RS(544,514) | 25GBASE-T |

### 3. 重传（Retransmission / ARQ）
| 机制 | 原理 |
|------|------|
| **停等 ARQ** | 发一包等 ACK |
| **回退 N (GBN)** | 出错包及之后全部重传 |
| **选择性重传 (SR)** | 只重传丢失的包（配合 SACK） |

## 各层差错控制详解

### L1 物理层 —— 比特级纠错（FEC）

**为什么 L1 要纠错？**
- 物理信号受噪声、衰减、色散影响，裸 BER 可达 10⁻⁶
- L2/L4 重传代价太大（RTT 级别），L1 在线纠错仅增加 ns 级延迟

| FEC 类型 | 说明 | 典型应用 |
|----------|------|----------|
| **RS-FEC (Reed-Solomon)** | 符号级纠错，100G 主流 | 100GBASE-DR4 / 400G |
| **KP4 FEC** | RS(544,514)，t=15 | 25GBASE-T 铜缆 |
| **BASE-R FEC (Clause 74)** | 轻量级，低延迟 | 早期 10G/25G |
| **卷积码 + Viterbi** | 流式编码 | Wi-Fi 802.11a/g/n |
| **LDPC** | 接近香农极限 | Wi-Fi 6、5G NR |
| **Turbo 码** | 4G 标准 | LTE |
| **交织 (Interleaving)** | 打散突发错误 | 配合 FEC，让突发错误变为随机错误 |

```
PAM-4 + FEC 流程:
数据 ──→ RS 编码 ──→ 交织 ──→ PAM-4 调制 ──→ 信道 ──→ PAM-4 解调 ──→ 解交织 ──→ RS 解码
原始 BER ~10⁻⁶                                                  纠错后 BER ~10⁻¹² ~ 10⁻¹⁵
```

> FEC 详解见 [osi/L1-physical.md 编码方式](../osi/L1-physical.md#fec前向纠错-现代高速链路的标配)

### L2 链路层 —— 帧级检错 + 可选重传

#### 以太网（Ethernet）—— 只检不纠
```
┌──────┬──────┬─────┬──────┬─────┬─────┐
│ DMAC │ SMAC │Type │ Data │ FCS │     │
│ 6B   │ 6B   │ 2B  │46-1500│ 4B │     │
└──────┴──────┴─────┴──────┴─────┘
                           ↑
                    CRC-32 校验

校验过程:
  发送方: FCS = CRC-32(DMAC+SMAC+Type+Data)
  接收方: 重算 CRC-32，与 FCS 对比
    相等 → 接受
    不等 → 静默丢弃（不重传，不通知发送方）

为什么以太网不重传？
  - 现代交换式以太网 BER 极低（<10⁻¹²）
  - 重传复杂度高，且需接收缓存
  - 上层（TCP）已提供可靠传输
```

#### Wi-Fi（802.11）—— 检错 + 单跳重传
```
Wi-Fi 在 L2 有 ARQ:
  发送方: 发送数据帧（含 FCS）
  接收方:
    CRC 正确 → 回复 ACK
    CRC 错误 → 不回复 ACK
  发送方: 超时未收到 ACK → 重传

  Wi-Fi 必须重传的原因:
  - 无线信道 BER 远高于有线（10⁻⁵ ~ 10⁻⁶）
  - 没有 TCP 也能保证单跳可靠（如管理帧）
  - 空口资源宝贵，丢包后上层重传代价太大
```

#### HDLC / PPP —— 完整 ARQ
```
HDLC 帧格式:
┌─────┬────────┬──────────┬──────┬─────┐
│Flag │Address │ Control  │ Data │ FCS │
│ 7E  │        │ (N(S)/N(R))│    │     │
└─────┴────────┴──────────┴──────┴─────┘
               ↑
        N(S): 发送序列号
        N(R): 期望收到的下一个序号（隐含 ACK）

ARQ 模式:
  - 停等: 发一帧等 ACK
  - 滑动窗口: 窗口内多帧并行，支持 GBN / SR
```

### L3 网络层 —— 仅校验头部 + ICMP 错误报告

#### IPv4 头部校验和
```
特点:
  - 仅校验 IP 头部（不含载荷！载荷靠 L4 校验和）
  - 每经过一个路由器重新计算（TTL 变化导致头部变化）
  - 算法: 16 bit 反码求和取反

为什么 IPv6 移除了头部校验和？
  - 上层（TCP/UDP）已有校验和
  - 下层（L2）已有 FCS
  - 每跳重算校验和浪费路由器性能
  - 三重保护冗余，移除一层无损可靠性
```

#### ICMP 错误报告
```
IP 本身不可靠，但通过 ICMP 向上层报告错误:
  - Destination Unreachable (Type 3): 目的不可达
  - Time Exceeded (Type 11): TTL 超时
  - Parameter Problem (Type 12): IP 头参数错误
  - Fragmentation Needed (Type 3, Code 4): 需分片但 DF=1（PMTUD 用）

注意: ICMP 错误消息本身不触发重传，重传由 L4（TCP）或应用层决策
```

### L4 传输层 —— 端到端差错控制的主战场

#### TCP —— 完整的可靠传输机制

| 机制 | 说明 | 解决的问题 |
|------|------|-----------|
| **校验和** | 16 bit（含伪头部） | 数据损坏检测 |
| **序列号** | 32 bit，按字节编号 | 有序交付、去重 |
| **累积 ACK** | 确认号 = 期望收到的下一个字节 | 确认收到 |
| **SACK** | 选择性确认已收到的非连续块 | 减少不必要重传 |
| **超时重传 (RTO)** | RTO = EstimatedRTT + 4×DevRTT | 丢失恢复 |
| **快速重传** | 3 个重复 ACK 立即重传 | 不等待超时 |
| **指数退避** | 超时后 RTO 翻倍 | 避免拥塞时雪崩 |

```
TCP 差错控制流程:
  发送方:
    发送段 (SEQ=x, 数据)
    启动 RTO 计时器
  接收方:
    校验和错误 → 静默丢弃
    校验和正确 → 发 ACK
  发送方:
    收到 ACK(ack=y) → 取消计时器，继续发送
    超时未收到 ACK → 重传该段
    收到 3 个重复 ACK → 快速重传丢失段
```

#### UDP —— 仅校验，不可靠
```
UDP 校验和:
  - 可选（IPv4 中可置 0，IPv6 强制）
  - 16 bit，含伪头部
  - 错误时静默丢弃，不重传

如果应用需要可靠:
  - 上层自己实现（QUIC、RUDP、DNS 重试）
  - 或改用 TCP
```

#### QUIC —— L4 重传 + 加密完整性
```
QUIC 差错控制:
  - AEAD 加密 (AES-GCM / ChaCha20-Poly1305)
    同时提供加密 + 完整性校验（替代独立校验和）
  - 包号 (Packet Number) 单调递增，不重用（防重放）
  - ACK 帧可携带多个 ACK Range（类似 SACK）
  - 丢包重传由 QUIC 自己实现（不依赖 UDP）
  - 无队头阻塞：不同 Stream 独立
```

### L7 应用层 —— 业务级校验与重试

| 协议 | 差错控制机制 |
|------|-------------|
| **HTTP** | 状态码（4xx/5xx 触发应用重试）、ETag + If-None-Match（幂等 GET） |
| **DNS** | UDP 5 秒超时重传；响应 > 512 字节转 TCP；DNSSEC 签名校验 |
| **FTP** | 控制连接 + 数据连接分离，MDTM/SIZE 校验文件完整性 |
| **SMTP** | 状态码（2xx/4xx/5xx）、队列重试、DKIM/SPF 防篡改 |
| **QUIC (HTTP/3)** | L4 QUIC 处理，应用层透明 |
| **文件校验** | MD5 / SHA-256 / CRC32 文件校验和（独立于协议） |
| **区块链** | 哈希指针 + 共识机制 |

```
应用级校验典型场景:
  下载文件:
    1. HTTP 下载（依赖 TCP 可靠传输）
    2. 下载后校验 SHA-256（与官网公布的 hash 对比）
    3. 不一致 → 重下（应用层重试）

  为什么还要应用级校验？
    - TCP 校验和仅 16 bit，强哈希才能抗恶意篡改
    - 检测中间件（CDN/代理）篡改
    - 端到端完整性证明（不信任中间任何节点）
```

## 各层差错控制对比总览

| 层级 | 检错机制 | 纠错/重传 | 保护范围 | 失败时行为 |
|------|---------|----------|----------|-----------|
| **L1** | FEC 编码冗余 | **在线纠错**（ns 级） | 物理比特 | 超出纠错能力 → 丢包 |
| **L2 有线** | CRC-32 | ❌ 不重传 | 单跳帧 | 静默丢弃 |
| **L2 无线** | CRC-32 + MIC | ✅ ACK + 重传 | 单跳空口 | 超时重传 |
| **L3** | 头部校验和（仅 IPv4） | ❌ 不重传 | IP 头 | ICMP 报告，丢包 |
| **L4 TCP** | 16 bit 校验和 | ✅ RTO + 快速重传 | 整个段（含伪头部） | 重传 + 拥塞控制 |
| **L4 UDP** | 16 bit 校验和（可选） | ❌ 不重传 | 整个段 | 静默丢弃 |
| **L4 QUIC** | AEAD 认证 | ✅ 重传 | 整个 QUIC 包 | 重传，无队头阻塞 |
| **L7** | 哈希/签名/状态码 | ✅ 应用重试 | 业务数据 | 取决于应用策略 |

## 关键设计原则

### 1. 端到端原则 (End-to-End Principle)
> 中间节点（L1-L3）只能尽力而为，**真正的可靠性必须由端点（L4 及以上）保证**。
>
> 理由：中间节点可能宕机、缓存丢失、恶意篡改，只有端点掌握完整业务语义。

### 2. 分层职责清晰
- **L1 做纠错**：物理错误实时修，避免上层感知
- **L2 做检错**：把已损坏的帧过滤掉，不让错误扩散
- **L3 只校验头部**：路由决策需要正确头部，数据完整性不管
- **L4 做可靠**：端到端重传，保证字节流正确有序
- **L7 做业务校验**：防篡改、防中间件破坏

### 3. 校验和 ≠ 加密认证
| 类型 | 抗随机错误 | 抗恶意篡改 | 典型场景 |
|------|-----------|-----------|----------|
| CRC / Checksum | ✅ | ❌ | L2 FCS、IP/TCP/UDP 校验和 |
| 密码学哈希 (SHA) | ✅ | ✅ | 文件完整性、证书 |
| HMAC / AEAD | ✅ | ✅ + 保密 | TLS、IPsec、MACsec |

### 4. 不同网络的策略差异

| 网络类型 | L1 FEC | L2 重传 | L4 重传 | 设计思路 |
|----------|--------|---------|---------|----------|
| **数据中心短距** | 轻量 FEC | ❌ | TCP | 低延迟优先 |
| **长距光传输** | 强 RS-FEC | ❌ | TCP | L1 纠错降低 RTT 代价 |
| **Wi-Fi** | 卷积/LDPC | ✅ ACK | TCP | 无线不可靠，L2 必须兜底 |
| **5G URLLC** | 强 FEC + HARQ | ✅ | (可跳过 L4) | 毫秒级延迟，L2 重传替代 L4 |
| **卫星** | 强 FEC + 交织 | 可选 | TCP 需优化 | RTT 长，TCP 慢启动吃亏 |
